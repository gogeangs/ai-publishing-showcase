# AI 퍼블리싱을 위한 디자이너 가이드라인

이 문서는 갈매기대도 랜딩페이지 AI 퍼블리싱 테스트에서 **실제 발생한 문제**를 기반으로 작성되었습니다.
디자이너가 아래 규칙을 지키면 AI 퍼블리싱의 시각적 유사도가 크게 올라갑니다.

---

## 테스트에서 발견된 디자이너 측 원인 문제들

테스트 과정에서 AI가 시안을 제대로 해석하지 못한 원인 중, **디자이너의 Figma 작업 방식**에서 비롯된 것들입니다:

| 문제 | 발생 상황 | 영향 |
|------|----------|------|
| 키카피 텍스트 오버라이드 미적용 | 모든 섹션 제목이 "100% 고객 만족 후기"로 동일하게 표시 | AI가 잘못된 텍스트로 퍼블리싱, 또는 임의 텍스트 생성 |
| 마스크 처리된 로고 | 로고가 마스크 레이어로 구성되어 MCP가 직접 추출 불가 | AI가 텍스트로 대체 ("갈매기대도") |
| 인터랙션 동작 미명시 | 슬라이더 방향, 탭 전환 방식, sticky 동작 등 미확인 5건 | AI가 정적 UI만 생성, 동작 로직 누락 |
| 업무요청서와 시안 불일치 | 푸터 배경색 — 문서: 흰색, 시안: #006343 | AI가 어떤 것을 따를지 혼란 |
| 특수 폰트 미처리 | 아치형 텍스트에 KoreanTBSMR (웹폰트 없음) 사용 | AI가 일반 텍스트로 렌더링, 시안과 전혀 다른 모습 |
| 복잡한 레이어 중첩 | 에셋이 여러 레이어에 걸쳐 조합되어 있음 | MCP 자동 추출 실패 → 플레이스홀더로 대체 |
| 장식 섹션 구분 불명확 | 배경이미지 섹션이 독립 콘텐츠가 아닌 장식처럼 보임 | AI가 해당 섹션 자체를 누락 (2/3 테스트에서 발생) |

## 규칙 1: 텍스트 오버라이드를 반드시 적용하라

### 이것이 가장 중요한 규칙인 이유

이번 테스트에서 **가장 큰 문제**였습니다.

시안의 모든 섹션 키카피가 "100% 고객 만족 후기"로 표시되어 있었습니다.
원인: 컴포넌트 인스턴스의 텍스트 오버라이드가 적용되지 않음.

```
[ 시안에서 보이는 것 ]                     [ 실제 필요한 텍스트 ]
차별점 섹션: "100% 고객 만족 후기"    →    "갈매기대도 만의 차별점"
상차림 섹션: "100% 고객 만족 후기"    →    "갈매기대도의 상차림"
창업절차 섹션: "100% 고객 만족 후기"  →    "창업절차"
문의하기 섹션: "100% 고객 만족 후기"  →    "문의하기"
```

### AI에 미치는 영향

- Figma MCP는 시안에 표시된 텍스트를 그대로 읽습니다
- 오버라이드가 안 되어 있으면, AI는 모든 섹션에 동일한 제목을 넣습니다
- 또는 "맥락에 맞게 판단해서 작성"하라는 지시를 받아 AI가 임의로 만듭니다
- 두 경우 모두 **시안과 다른 텍스트**가 들어갑니다

### 디자이너가 해야 할 것

```
□ 컴포넌트 인스턴스를 사용했다면, 각 인스턴스의 텍스트 오버라이드를 확인
□ 모든 섹션의 키카피(제목)가 해당 섹션에 맞는 텍스트인지 확인
□ 서브카피(부제목), 본문 텍스트도 실제 내용으로 채우기
□ 시안에 "여기에 텍스트 입력" 같은 플레이스홀더 텍스트가 남아있지 않은지 확인
```

### 확인 방법

Figma에서 시안을 위에서 아래로 스크롤하면서:
- 각 섹션의 제목이 **서로 다른지** 확인
- 본문 텍스트가 **실제 내용**인지 확인 (Lorem ipsum 아닌지)
- 테이블/카드의 데이터가 **실제 데이터**인지 확인

## 규칙 2: 에셋은 Flatten 또는 Export Ready 상태로 만들어라

### 발생한 문제

로고 에셋이 **마스크 레이어**로 구성되어 있어서 MCP가 직접 추출할 수 없었습니다.

```
[ 시안 레이어 구조 ]
📁 가로 로고 (흰색)
  📁 마스크 그룹
    🔲 직사각형 (클리핑 영역)
    📝 로고 벡터 패스 1
    📝 로고 벡터 패스 2
    📝 로고 벡터 패스 3
```

이렇게 복잡하게 중첩된 구조는 MCP가 **단일 이미지로 인식하지 못합니다**.
결과: AI가 `<img>` 태그 대신 텍스트 "갈매기대도"로 대체 (test-1, test-2 모두).

### 디자이너가 해야 할 것

```
□ 로고, 아이콘 등 Export될 에셋은 Flatten(병합) 처리
   → 마스크, 부울 연산 결과를 하나의 레이어로 병합

□ 또는 Export 설정을 미리 지정
   → 해당 레이어 선택 → Design 패널 → Export → SVG 또는 PNG 추가
   → MCP가 Export 설정이 있는 레이어를 우선 인식합니다

□ 벡터 에셋(로고, 아이콘)은 SVG Export 설정
□ 사진 에셋(음식, 매장)은 PNG 또는 JPG Export 설정
□ 복잡한 구성(아치형 텍스트, 붓터치+텍스트 조합)은 통이미지로 Export 설정
```

### 에셋별 권장 형식

| 에셋 유형 | 권장 형식 | 이유 |
|-----------|----------|------|
| 로고, 아이콘 | **SVG** | 벡터라서 해상도 독립적, 파일 작음 |
| 사진 (음식, 매장) | **JPG/WebP** | 용량 최적화 |
| 투명 배경 이미지 (누끼) | **PNG** | 투명도 필요 |
| 특수 폰트 텍스트 | **SVG** (통이미지) | 웹폰트 없는 서체는 이미지 대체 |
| 패턴 타일 | **PNG** | CSS background-repeat로 사용 |
| 복합 구성 (텍스트+장식) | **SVG** (통이미지) | 레이어 분리하면 재현 불가 |

## 규칙 3: Figma Variables(변수)를 사용하라

### 발생한 문제와 효과

test-3에서 `get_variable_defs`로 디자인 토큰을 추출했을 때,
Figma Variables에 정의된 색상이 CSS 변수로 정확하게 매핑되었습니다:

```
Figma Variable          →    CSS Variable
MainGreen: #006343      →    --main-green: #006343
MainOrange: #EC6937     →    --main-orange: #EC6937
본문컬러: #505050        →    --body-color: #505050
Black: #111111          →    --black: #111111
```

반면 Variables를 사용하지 않고 직접 색상을 입힌 요소는,
AI가 하드코딩된 hex 값을 써서 나중에 색상 변경이 어려워집니다.

### 디자이너가 해야 할 것

```
□ 프로젝트에서 사용하는 색상을 Figma Variables로 정의
   → 브랜드 주색(MainGreen, MainOrange 등)
   → 텍스트 색상(Black, 본문컬러, Muted 등)
   → 배경 색상(Beige, Ivory, LightGreen 등)
   → 강조/경고 색상(Red 등)

□ 모든 색상을 직접 입력(#006343) 하지 말고 Variable로 적용
   → 시안의 모든 요소가 Variable을 참조하도록

□ 폰트 크기도 Variables로 정의하면 더 좋음
   → Heading1: 52px, Heading2: 42px, Body: 17px 등

□ 간격(spacing)도 Variables로 관리하면 최적
   → section-padding-pc: 100px, section-padding-mo: 60px 등
```

### Variable 네이밍 권장

AI가 CSS 변수명으로 변환하므로, **영어 + 용도 기반** 네이밍을 권장합니다:

```
권장:
  color/main-green      → --color-main-green
  color/main-orange     → --color-main-orange
  color/body            → --color-body
  bg/beige              → --bg-beige
  text/heading-1        → --text-heading-1

비권장:
  본문컬러               → --본문컬러 (CSS에서 한글 변수명은 가능하지만 불편)
  색상1                  → --색상1 (의미 파악 불가)
  #006343직접입력        → (Variable 미사용)
```

## 규칙 4: 프레임 구조를 명확하게 정리하라

### 발생한 문제

AI가 "배경이미지" 섹션을 2/3 테스트에서 누락한 이유:
- 해당 섹션이 시안에서 독립적인 콘텐츠 영역으로 보이지 않고, **전환 장식**처럼 보였음
- 프레임 이름이 불명확하거나, 다른 섹션과 구분이 어려웠을 가능성

### 디자이너가 해야 할 것

#### 프레임 네이밍 규칙

```
권장:
  📁 PC (1560px)
    📁 Section 01 - GNB
    📁 Section 02 - Hero (메인비주얼)
    📁 Section 03 - Brand (브랜드소개)
    📁 Section 04 - Background (배경이미지)       ← 번호가 있어서 AI가 "이것도 섹션"으로 인식
    📁 Section 05 - Difference (차별점)
    📁 Section 06 - Menu (상차림)
    📁 Section 07 - Reviews (고객후기)
    📁 Section 08 - Banner (중간배너)
    📁 Section 09 - Franchise Process (창업절차)
    📁 Section 10 - Franchise Cost (창업비용)
    📁 Section 11 - Contact (문의하기)
    📁 Section 12 - Footer (푸터)

비권장:
  📁 Frame 1560
    📁 상단
    📁 내용1
    📁 bg_img                  ← "섹션"이 아닌 "배경"으로 인식될 수 있음
    📁 내용2
    📁 하단
```

#### 핵심 원칙

```
□ 모든 섹션에 번호를 매긴다 (Section 01, 02, 03...)
   → AI가 번호를 보고 "독립적인 섹션"으로 인식

□ 한글 이름 + 영문 이름을 병기한다
   → MCP가 영문 이름을 CSS class/id로 사용

□ 장식 섹션도 독립 프레임으로 분리한다
   → 전환 배경, 중간 배너 등도 별도 섹션으로 명확히 구분

□ PC와 Mobile 프레임을 같은 페이지에 나란히 배치한다
   → MCP가 두 프레임을 비교하여 반응형 대응을 추론
```

#### 모바일 프레임 필수 포함 항목

```
□ Mobile 전체 페이지 프레임 (375px)
□ GNB 열림 상태 프레임 (햄버거 메뉴 오픈)     ← 이번 테스트에서 있었고, AI가 잘 활용함
□ (있으면 좋음) 모바일 드로어/모달 열린 상태
```

## 규칙 5: 인터랙션/동작을 시안에 명시하라

### 발생한 문제

이번 테스트에서 디자이너 확인이 필요했지만 **미확인으로 남은 항목이 5건**이었습니다:

| 미확인 항목 | AI의 대응 |
|------------|----------|
| 하단 문의 바 sticky 여부 | AI가 판단 불가 → 고정 없이 구현 |
| 상차림 탭 전환 시 애니메이션 | 탭 활성 스타일만 구현, 콘텐츠 교체 없음 |
| 문의 폼 제출 시 동작 | 기본 폼만 구현, 유효성 검증 없음 |
| GNB 스크롤 시 배경색 변경 | AI가 임의로 추가 (다행히 맞음) |
| 타겟 브라우저 범위 | AI가 모든 브라우저용으로 구현 |

### 디자이너가 해야 할 것

정적 시안에 표현할 수 없는 **동작(인터랙션)**을 명시해야 합니다.

#### 방법 1: Figma 코멘트/메모 (최소한)

시안에 직접 메모를 남깁니다:
```
[메인비주얼] 배경: 자동재생 비디오 (muted, loop)
[상차림] 탭 클릭 → 카드 콘텐츠 교체 (fade 전환), Swiper 좌우 슬라이드
[고객후기] 스크롤 시 리뷰 하나씩 왼쪽 슬라이드 + 로딩바 효과
[문의 폼] 제출 시 유효성 검증 → 서버 전송
[하단 바] 스크롤 시 sticky 표시
[GNB] 스크롤 시 배경 흰색 + 그림자 전환
```

이번 시안에서도 고객후기 섹션에 "스크롤 시 리뷰 하나씩 왼쪽으로 슬라이드 + 로딩바 효과"라는 메모가 있었고, 이 메모가 있었기 때문에 test-2에서 CSS 자동스크롤을 독자적으로 구현하는 성과가 있었습니다.

#### 방법 2: 인터랙션 스펙 시트 (권장)

시안과 별도로 인터랙션 동작을 정리한 프레임/페이지를 만듭니다:

```
[ 인터랙션 스펙 ]

■ GNB
  - 스크롤 100px 이상: 배경 흰색 + box-shadow
  - 메뉴 클릭: smooth scroll로 해당 섹션 이동
  - 활성 메뉴: 현재 보이는 섹션에 해당하는 메뉴 하이라이트

■ 메인비주얼
  - 배경: 비디오 자동재생 (muted, loop, playsinline)
  - 하단 바: 폼 입력 → 제출

■ 상차림
  - 탭: 구이류/곁들임 클릭 시 카드 콘텐츠 교체
  - 카드: Swiper 슬라이더 (좌우 드래그 + 화살표 버튼)
  - PC: 4개 표시 / Mobile: 1.2개 표시 (일부 노출)

■ 고객후기
  - 스크롤 시 카드가 왼쪽으로 자동 슬라이드
  - GSAP ScrollTrigger 사용
  - 로딩바: 스크롤 진행도 표시

■ 모바일
  - 햄버거 클릭: 풀스크린 메뉴 열기
  - X 클릭 또는 메뉴 항목 클릭: 메뉴 닫기
  - 메뉴 열린 동안 body 스크롤 잠금

■ 폼
  - 필수 필드: 성함, 휴대폰번호, 동의 체크
  - 유효성 검증: 빈 필드 알림
  - 제출: 서버 전송 + reCAPTCHA v3

■ 하단 문의 바
  - 메인비주얼 아래로 스크롤 시 화면 하단에 sticky 표시
  - 푸터 도달 시 숨김
```

## 규칙 6: 이미지 에셋의 크기를 최적화하라

### 발생한 문제

MCP가 자동 추출한 이미지 중 **비정상적으로 큰 파일**이 있었습니다:

| 파일 | 추출 용량 | 적정 용량 |
|------|:--------:|:--------:|
| hero-bg.png | **5.1MB** | ~200KB (JPG 압축) |
| diff-store.png | **5.1MB** | ~300KB (JPG 압축) |
| brand-badge.png | **3.0MB** | ~100KB (JPG 압축) |
| waiting-left.png | **2.4MB** | ~150KB (JPG 압축) |

원인: MCP가 원본 해상도(Figma 내부 해상도)로 추출. 웹용 최적화 없음.

### 디자이너가 해야 할 것

```
□ Figma에서 Export 설정 시 적절한 크기 지정
   → 사진: 최대 너비 1920px (2x 레티나 기준)
   → 카드 이미지: 실제 표시 크기의 2배 (290px 표시 → 580px Export)
   → 아이콘: 실제 크기의 2~3배

□ 사진은 JPG (품질 80%) 또는 WebP 형식 권장
   → PNG로 Export하면 불필요하게 용량이 큼

□ 로고/아이콘은 SVG 형식
   → 벡터는 항상 SVG가 최적
```

하지만 **디자이너가 이 부분을 놓쳐도 괜찮습니다**.
개발자가 추출된 에셋을 ImageMagick, squoosh.app 등으로 후처리하면 됩니다.
다만 디자이너가 미리 최적화하면 개발자의 후처리 시간을 줄일 수 있습니다.

## 규칙 7: 호버/활성/비활성 상태를 모두 보여줘라

### 발생한 문제

AI는 시안에 **보이는 상태만** 구현합니다:
- 탭의 "활성" 상태만 있고 "비활성" 상태가 없으면, AI가 비활성 스타일을 임의로 만듬
- 버튼의 hover 효과가 없으면, AI가 기본적인 opacity 변경만 적용
- GNB 스크롤 전/후 상태가 없으면, AI가 추측으로 구현

### 디자이너가 해야 할 것

```
□ 버튼: Default / Hover / Active / Disabled 상태
□ 탭: 활성 탭 + 비활성 탭 (둘 다 명확하게)
□ 카드: Default / Hover 상태
□ 인풋: Default / Focus / Error / Filled 상태
□ GNB: 투명 상태 (히어로 위) + 흰색 상태 (스크롤 후)
□ 모바일 메뉴: 닫힌 상태 + 열린 상태 (별도 프레임)
```

이번 테스트에서 **모바일 GNB 열림 상태**가 별도 프레임(노드 3:167)으로 있었고,
이 덕분에 3개 테스트 모두 모바일 햄버거 메뉴를 정확하게 구현할 수 있었습니다.
→ **상태별 프레임이 있으면 AI 구현 정확도가 올라갑니다.**

## 규칙 8: 시안과 문서의 일관성을 유지하라

### 발생한 문제

업무요청서와 Figma 시안이 **서로 다른 정보**를 가지고 있었습니다:

| 항목 | 업무요청서 | Figma 시안 |
|------|----------|-----------|
| 푸터 배경색 | "흰색 또는 밝은 회색" | **#006343 (MainGreen)** |
| 세로 로고 크기 | 127x140px | **약 61x67px** |

test-1 프롬프트에서 "충돌 시 업무요청서 우선"이라는 지시를 했지만,
이런 불일치가 많을수록 AI의 혼란이 커지고, 어느 쪽을 따르든 한쪽은 틀리게 됩니다.

### 디자이너가 해야 할 것

```
□ 기획서/업무요청서와 시안의 내용이 일치하는지 최종 확인
   → 색상, 폰트 크기, 레이아웃 수치

□ 불일치가 있으면 시안을 최종 버전으로 업데이트
   → 또는 업무요청서에 "시안이 최종"이라고 명시

□ 변경 이력이 있으면 시안에 반영 완료 후 핸드오프
   → "중간 버전"의 시안을 넘기지 않기
```

## 규칙 9: 웹폰트 없는 서체는 이미지로 대체 표시하라

### 발생한 문제

브랜드 섹션의 아치형 텍스트에 **KoreanTBSMR** 폰트가 사용되었으나,
이 폰트는 웹폰트로 제공되지 않습니다.

결과: AI가 일반 Pretendard 텍스트로 렌더링 → 시안의 아치형 캘리그래피와 전혀 다른 모습.

### 디자이너가 해야 할 것

```
□ 웹폰트로 제공되지 않는 서체를 사용한 요소를 파악

□ 해당 요소를 SVG 통이미지로 Export 설정
   → Figma에서 해당 텍스트 레이어 선택 → Outline Stroke (또는 Flatten)
   → Export 설정: SVG 추가

□ 또는 해당 레이어 이름에 "SVG-Export" 등 표시
   → 개발자/AI가 이 요소는 텍스트가 아닌 이미지임을 인지

□ 시안에 메모: "이 텍스트는 웹폰트 없음 → SVG 이미지로 대체"
```

### 웹폰트 제공 여부 확인 방법

| 폰트 | 웹폰트 | 처리 |
|------|:------:|------|
| Pretendard | O (CDN) | 그대로 사용 |
| GmarketSans | O (CDN) | 그대로 사용 |
| Paperlogy | O (CDN) | 그대로 사용 |
| KoreanTBSMR | **X** | **SVG 이미지로 대체** |
| 손글씨/캘리그래피 | 대부분 X | **SVG 이미지로 대체** |

## 부록: 기존 방식 vs AI 퍼블리싱에서 달라지는 점

디자이너 입장에서 기존 핸드오프와 AI 퍼블리싱용 핸드오프의 차이:

| 항목 | 기존 (사람 퍼블리셔) | AI 퍼블리싱 |
|------|-------------------|-----------|
| 텍스트 오버라이드 | 빠져있어도 퍼블리셔가 물어봄 | **AI는 안 물어보고 그대로 구현** |
| 에셋 Export | 퍼블리셔가 직접 Export | **MCP가 자동 추출 (설정 필요)** |
| 마스크/복잡 레이어 | 퍼블리셔가 판단하여 처리 | **MCP가 인식 못하면 누락** |
| 인터랙션 동작 | 구두로 설명 가능 | **시안에 명시 필요 (메모/스펙)** |
| 불명확한 부분 | 퍼블리셔가 질문 | **AI가 임의 판단** |
| 색상 관리 | 스포이드로 추출 | **Variables → CSS 변수 자동 매핑** |
| 폰트 이슈 | 퍼블리셔가 대체 폰트 판단 | **AI는 없으면 무시하거나 대체** |
| 섹션 누락 | 퍼블리셔가 발견하고 질문 | **AI가 장식 섹션을 건너뛸 수 있음** |

### 핵심 차이

> 기존에는 "대충 만들어도 퍼블리셔가 알아서 해석"했지만,
> AI 퍼블리싱에서는 **"시안에 있는 그대로"** 구현됩니다.
>
> 따라서 시안의 완성도가 곧 결과물의 완성도입니다.
