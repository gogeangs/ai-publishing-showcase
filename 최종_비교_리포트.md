## 핵심 인사이트

### 1. 에셋이 시각적 유사도의 60%를 결정한다

test-3과 test-4가 상위인 가장 큰 이유는 Figma Desktop Local MCP가 **이미지 에셋을 자동 추출**했기 때문입니다.

```
레퍼런스 화면에서 이미지가 차지하는 비중: 약 60%
→ 이미지가 있는 test-3/4: 레퍼런스와 "같은 느낌"
→ 이미지가 없는 test-1/2: "와이어프레임" 느낌
```

### 2. 업무요청서v2 + MCP 조합이 최적의 결과를 낸다

test-4는 **업무요청서v2(텍스트/인터랙션 명세) + MCP(에셋/디자인 토큰 추출)**를 결합하여,
test-1의 강점(구조 완성도)과 test-3의 강점(에셋)을 동시에 달성했습니다.

| | 구조 완성도 | 에셋 | 인터랙션 | 시각적 유사도 |
|---|:---:|:---:|:---:|:---:|
| test-1 (상세 스펙) | **12/12** | 0개 | X | 약 35% |
| test-3 (MCP 에셋추출) | 11/12 | **32개** | X | 약 65% |

> 에셋 + 구조 + 인터랙션을 모두 갖춘 test-4가 가장 높은 유사도를 달성했습니다.

### 3. 업무요청서v2는 "MCP가 못 읽는 것만" 작성하는 것이 핵심

test-1의 업무요청서v1은 893줄로 색상/간격/크기까지 모두 포함해 MCP와 50건 이상 충돌이 발생했습니다.
test-4의 업무요청서v2는 327줄로 텍스트, 인터랙션, 디자이너 결정사항만 포함하여 충돌을 원천 차단했습니다.
프롬프트 자동작성으로 준비 시간도 2~4시간 → 25~40분으로 단축되었습니다.

### 4. test-4는 인터랙션의 돌파구를 열었다

test-1~3은 모두 슬라이더, 스크롤 애니메이션이 미구현이었으나,
test-4는 **커스텀 슬라이더(transform), IntersectionObserver fadeUp, 폼 검증 3곳**을 구현했습니다.
Swiper/GSAP 수준은 아니지만, "정적 페이지"에서 "반응하는 페이지"로 한 단계 진화했습니다.

### 5. Figma URL만으로도 기본 구조는 잡힌다

test-2는 Figma URL 하나만 주었지만 기본 구조를 모두 만들어냈습니다. 준비 시간 대비 결과가 나쁘지 않습니다.

### 최적 워크플로우 제안 (워크플로우 C)

```
[1단계] PM: 업무요청서v2 작성 (프롬프트로 자동작성, 25~40분)
   ↓   텍스트, 인터랙션, 디자이너 결정사항만 정리
   ↓
[2단계] AI: 업무요청서v2 + Figma MCP로 1차 퍼블리싱
   ↓   이미지 에셋 자동 추출 + 12/12 섹션 + 기본 인터랙션
   ↓
[3단계] 퍼블리셔: 인터랙션 보강 (사람 작업)
   ↓   Swiper.js 교체, GSAP/AOS 정교화, 비디오 배경
   ↓

   ↓   태블릿 중간 브레이크포인트 추가, 세부 간격 조정
   ↓
```

- 에셋 수급 시간을 **MCP 자동화**로 크게 단축 (가장 큰 병목 제거)
- 업무요청서v2로 **텍스트 정확도 + 인터랙션 구현**까지 확보
- AI가 만든 **깔끔한 코드 구조** 위에 사람이 인터랙션만 추가
- 처음부터 CSS 변수 체계가 잡혀 있어 **디자인 토큰 변경이 쉬움**
